# 专利特征分析框架

> 本文件是 sptler 在查新/FTO/无效等场景进行技术特征拆解与检索内容准备的专业方法。
> 核心思想：**把技术方案拆成可检索的要素，每个要素展开成可执行的检索词，最终构建可跑的检索式。**
> **下游检索引擎**：OpenSearch（语义检索 + 关键词检索双路）。本框架的产出须同时适配两条路。

## 一、三要素拆解法（问题-方案-效果）

任何技术方案都可拆为三个层面，每个层面提取可检索特征：

### 1. 技术问题（要解决什么）
- 这个发明要解决什么现有技术的缺陷？
- 用一句话描述"现有技术的问题是……"
- 提取关键词：问题本身 + 缺陷描述

**示例**：蓝牙定位精度不够（RSSI信号衰减导致距离估算误差大）
- 关键词：定位精度、信号衰减、距离估算、误差

### 2. 技术方案（怎么解决）
这是核心——逐特征拆解：

| 特征编号 | 特征描述 | 特征类型 | 必要性 | 检索权重 |
|---|---|---|---|---|
| A | {核心特征} | 结构/方法/参数 | 必要（独立权） | 高（必须命中） |
| B | {辅助特征} | 结构/方法/参数 | 必要（独立权） | 高（必须命中） |
| C | {优选特征} | 结构/方法/参数 | 从属（可收窄） | 中（提升相关性） |
| D | {替代特征} | 结构/方法/参数 | 从属（替代实施例） | 低（扩展召回） |

特征类型说明：
- **结构特征**：部件、连接关系、位置、形状、材料
- **方法特征**：步骤、顺序、条件、参数
- **参数特征**：数值范围、比例、阈值

> **检索权重**：告诉下游 OpenSearch 哪个特征重要——高权重特征命中=强相关，低权重命中=弱相关。这是 sptler 比裸 Claude 多的一层：不只拆特征，还标优先级。

### 3. 技术效果（解决了什么效果）
- 这个方案带来什么技术效果？
- 用一句话描述"由此实现了……"
- 提取关键词：效果描述 + 量化指标

**示例**：通过AOA到达角测量替代RSSI，定位精度提升至厘米级
- 关键词：到达角、AOA、厘米级、精度提升

## 二、双路语义扩展（关键词路 + 语义路）

> **关键改进**：每个特征须同时产出两路检索内容——
> - **关键词路**（BM25 精确匹配）：同义词、近义词、英文、分类号
> - **语义路**（向量检索）：自然语言概念描述、功能叙述、问题-效果对

### 关键词路（传统，精确匹配）

| 扩展维度 | 说明 | 示例（以"到达角"为例） |
|---|---|---|
| **同义词** | 不同说法表达同一概念 | 到达角、入射角、angle of arrival、AoA |
| **近义词** | 概念相近但不完全相同 | 到达方向、来波方向、方向角 |
| **上下位** | 更宽/更窄的概念 | 上位：角度测量；下位：MUSIC算法估计AoA |
| **英文对照** | 中文特征对应的英文术语 | angle of arrival, AoA, direction of arrival, DOA |
| **IPC/CPC分类号** | 该特征对应的专利分类号 | G01S 5/02（利用无线电波定位）、G01S 5/04 |
| **功能表达** | 不用结构词，用功能描述 | 测量信号到达方向、确定信号来源方位 |

### 语义路（向量检索，概念匹配）

> 语义路不是关键词列表，而是**自然语言描述**——让 OpenSearch 的向量模型能"理解"这个特征在说什么。

| 语义维度 | 说明 | 示例（以"AOA角度估计"为例） |
|---|---|---|
| **概念描述** | 一句话说清这个特征在做什么 | "通过测量无线电信号到达天线阵列的角度来估计信号源方向" |
| **问题-效果对** | 这个特征解决什么问题+带来什么效果 | "问题：RSSI信号衰减导致距离估算不准 → 效果：用角度测量替代信号强度，精度提升至厘米级" |
| **类比表达** | 用不同领域的类比描述同一功能 | "类似声源定位中通过麦克风阵列判断声音方向" |
| **否定边界** | 这个特征**不是**什么（帮 OpenSearch 排除噪声） | "不是RSSI信号强度测距，不是飞行时间ToF测距，不是载波相位定位" |

> **为什么语义路重要**：关键词路能找到"说了到达角"的专利；语义路能找到"没说到达角但描述了同样的角度测向原理"的专利。**后者是裸 Claude 检索不到的**——因为裸 Claude 只会想同义词，不会构造概念描述+否定边界。

### 扩展方法

**关键词路**：
1. **同义词**：问"这个特征还能怎么说？"——换主语、换动词、换名词
2. **近义词**：问"有什么概念跟它接近但不一样？"——区分边界
3. **上下位**：问"它的上位概念是什么？它的具体实现有哪些？"——上位扩检索范围，下位缩范围
4. **英文对照**：问"英文专利里这个词怎么写？"——必须查英文，因为大量对比文件是英文
5. **分类号**：问"这个技术领域对应哪个IPC/CPC分类号？"——用分类号锁定技术领域
6. **功能表达**：问"不用具体结构，用功能怎么描述？"——防止对手用不同结构实现同样功能

**语义路**：
1. **概念描述**：问"如果不用任何专业术语，用大白话怎么说这个特征在做什么？"
2. **问题-效果对**：问"这个特征为什么存在？没有它会怎样？有了它怎样？"
3. **类比表达**：问"其他领域有没有做同样事情的不同方法？"
4. **否定边界**：问"容易跟这个特征混淆但实际不同的技术有哪些？"——**这是 sptler 的独家差异化**，裸 Claude 不会主动想"不是什么"，但否定边界能让 OpenSearch 语义检索精准排除噪声。

## 三、检索式构建方法（双路产出）

### 关键词路检索式（给 BM25 用）

#### 1. 核心检索式（先跑这个）
```
(特征A的同义词集合) AND (特征B的同义词集合) AND 技术领域分类号
```

**示例**：
```
(到达角 OR AoA OR angle of arrival OR 方向角) 
AND (蓝牙 OR Bluetooth OR 蓝牙定位) 
AND (IPC=G01S5/02 OR CPC=G01S5/02)
```

#### 2. 扩展检索式（核心无结果时跑）
```
(特征A的上位概念) AND (特征B的功能表达) AND 技术领域
```

**示例**：
```
(角度测量 OR 方向测量) 
AND (短距离无线定位 OR 室内定位) 
AND (IPC=G01S5/00)
```

#### 3. 否定过滤（降噪）
```
核心检索式 NOT (否定边界词)
```

**示例**：
```
(到达角 OR AoA) AND (蓝牙) NOT (RSSI OR 信号强度 OR 载波相位)
```

> **否定过滤是 sptler 的独家**：裸 Claude 不会主动构造 NOT 表达式。但 OpenSearch 的 BM25 支持 NOT，能大幅降低噪声——比如查 AOA 定位时排除所有 RSSI 专利。

### 语义路检索内容（给向量检索用）

> 语义路不是布尔表达式，是**自然语言查询文本**——直接作为向量检索的 query。

#### 1. 语义核心查询
```
{概念描述} + {问题-效果对}
```

**示例**：
```
通过测量无线电信号到达天线阵列的角度来估计信号源方向的方法，
用于解决RSSI信号衰减导致距离估算不准的问题，实现厘米级定位精度。
```

#### 2. 语义扩展查询（用类比拓宽召回）
```
{类比表达} 在专利中的应用
```

**示例**：
```
类似声源定位中通过麦克风阵列判断声音方向的方法，应用于无线通信领域的信号方向估计
```

#### 3. 语义否定边界（帮向量模型排除相似但不同的专利）
```
检索"{概念描述}"但排除"{否定边界描述}"
```

**示例**：
```
检索"无线电信号角度测向定位"但排除"基于信号强度RSSI的测距方法"和"基于飞行时间ToF的测距方法"
```

> **语义路+关键词路双跑**：把关键词检索式的命中 + 语义查询的命中合并去重——这是比单一检索路多出来的召回。**sptler 产出双路内容，下游 OpenSearch 双路跑，合并结果——这是 sptler 比裸 Claude 强的核心：裸 Claude 只产关键词，不产语义查询文本。**

### 4. 补充检索策略
- **引文追踪**：找到1-2篇高相关对比文件后，追踪其引用/被引用
- **同族追踪**：查对比文件的同族（其他国家的对应申请）
- **发明人追踪**：查该领域核心发明人的其他申请
- **申请人追踪**：查竞争对手的专利组合

### 5. 检索完备性自检
完成检索准备后，邹蕴组织自检：
- [ ] 每个必要特征是否都有**关键词路**检索词覆盖？
- [ ] 每个必要特征是否都有**语义路**概念描述+否定边界？
- [ ] 中英文是否都检索了？
- [ ] 上位概念是否检索了？
- [ ] 分类号是否覆盖了？
- [ ] 否定边界是否构造了（排除噪声）？
- [ ] 是否有特征被遗漏（只检索了A忘了B）？
- [ ] 语义查询文本是否自然通顺（向量模型能理解）？
- [ ] 高相关对比文件是否做了引文追踪？

## 四、角色分工（按能力而非固定圣人）

特征分析框架是方法论，不绑定特定圣人。具体由 route_sages 按议题动态路由的入会圣人承担以下角色：

| 角色 | 能力要求 | 职责 |
|---|---|---|
| **特征拆解者** | 擅长结构化拆解技术方案 | 三要素拆解：技术方案→A/B/C/D特征，标注必要性+类型 |
| **边界确认者** | 擅长权利要求边界精修 | 每个特征的边界范围（宽到什么程度、窄到什么程度） |
| **语义扩展者** | 擅长数据语义/检索/分类 | 每个特征展开同义词/近义词/上下位/英文/分类号，构建检索式 |

> 哪位圣人担任哪个角色，由 route_sages 按议题关键词动态路由决定，不在方法论里写死。模板中的"责任人"字段由实际入会圣人填充。

## 五、与其他场景的关系

- **查新检索**：特征拆解→检索准备→用户提供对比文件→逐特征新颖性比对→创造性判断→结论
- **FTO检索**：特征拆解→用户提供风险专利→逐特征侵权比对（全要件+等同）→规避设计
- **无效宣告**：特征拆解→用户提供目标专利→逐特征找无效理由（新颖性/创造性/公开不充分/不支持）
- **OA答复**：特征拆解→审查员引用对比文件→逐特征找区别技术特征→答复策略

特征分析是所有专利场景的共同基础能力。
