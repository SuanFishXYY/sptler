# 专利特征分析框架

> 本文件是 sptler 在查新/FTO/无效等场景进行技术特征拆解与检索内容准备的专业方法。
> 核心思想：**把技术方案拆成可检索的要素，每个要素展开成可执行的检索词，最终构建可跑的检索式。**
> **下游检索引擎**：OpenSearch（语义检索 + 关键词检索双路）。本框架的产出须同时适配两条路。

## 一、三要素拆解法（问题-方案-效果）

任何技术方案都可拆为三个层面，每个层面提取可检索特征：

### 1. 技术问题（要解决什么）
- 这个发明要解决什么现有技术的缺陷？
- 用一句话描述"现有技术的问题是……"
- 提取关键词：问题本身 + 缺陷描述

**示例**：蓝牙定位精度不够（RSSI信号衰减导致距离估算误差大）
- 关键词：定位精度、信号衰减、距离估算、误差

### 2. 技术方案（怎么解决）
这是核心——逐特征拆解：

| 特征编号 | 特征描述 | 特征类型 | 必要性 | 检索权重 | 依赖关系 |
|---|---|---|---|---|---|
| A | {核心特征} | 结构/方法/参数 | 必要（独立权） | 高（必须命中） | 独立（A可单独存在） |
| B | {辅助特征} | 结构/方法/参数 | 必要（独立权） | 高（必须命中） | 依赖A（B必须与A共存） |
| C | {优选特征} | 结构/方法/参数 | 从属（可收窄） | 中（提升相关性） | 依赖A+B |
| D | {替代特征} | 结构/方法/参数 | 从属（替代实施例） | 低（扩展召回） | 替代B（D是B的另一种实现） |

特征类型说明：
- **结构特征**：部件、连接关系、位置、形状、材料
- **方法特征**：步骤、顺序、条件、参数
- **参数特征**：数值范围、比例、阈值

**依赖关系**（影响检索式组合逻辑）：
- **独立特征**：可单独检索（A OR A的同义词）
- **依赖特征**：必须与被依赖特征组合检索（A AND B，不能 B 单独）
- **替代特征**：与原特征互斥（B OR D，不要求同时命中）
- **组合特征**：A+B 构成一个功能单元，应作为整体语义描述

> **依赖关系是 sptler 比裸 Claude 多的一层**：裸 Claude 拆特征但不标依赖，检索式可能把依赖特征单独检索导致误召回。标了依赖关系后，检索式组合逻辑更准确——独立特征分别召回，依赖特征组合召回，替代特征 OR 展开。

#### 多实施例拆解

一个特征可能有多种实现方式（实施例），每种需单独覆盖检索：

| 特征 | 实施例1 | 实施例2 | 实施例3 |
|---|---|---|---|
| A（角度估计） | MUSIC算法 | ESPRIT算法 | 波束形成法 |
| B（天线阵列） | 均匀线阵 | 均匀圆阵 | L型阵列 |

> 每个实施例都需要在语义扩展中覆盖——否则只检索单一算法的专利，漏掉用其他算法实现同样功能的对比文件。**这是漏检的常见原因**：检索者只想到一种实现，没想到竞品可能用另一种。

> **检索权重**：告诉下游 OpenSearch 哪个特征重要——高权重特征命中=强相关，低权重命中=弱相关。这是 sptler 比裸 Claude 多的一层：不只拆特征，还标优先级。

### 3. 技术效果（解决了什么效果）
- 这个方案带来什么技术效果？
- 用一句话描述"由此实现了……"
- 提取关键词：效果描述 + 量化指标

**示例**：通过AOA到达角测量替代RSSI，定位精度提升至厘米级
- 关键词：到达角、AOA、厘米级、精度提升

## 二、等同特征簇（同功能不同结构）

> **这是 sptler 比裸 Claude 最核心的差异化**。裸 Claude 会找同义词（换个说法说同一件事），但不会找**等同技术方案**（换种结构实现同一功能）。FTO 等同侵权判定、无效检索召回、语义检索拓宽——都靠这一步。

对每个必要特征，问：**"还有什么不同的技术手段能实现同样功能？"**

| 特征 | 功能 | 等同技术方案1 | 等同技术方案2 | 等同技术方案3 |
|---|---|---|---|---|
| A（AOA角度估计） | 测量信号到达方向 | MUSIC算法（子空间分解） | ESPRIT算法（旋转不变） | 波束形成扫描（逐步移相） |
| B（天线阵列） | 接收多路信号 | 均匀线阵 | 均匀圆阵 | 平面阵列 |

等同特征簇的用途：
- **FTO 侵权**：等同原则——被告用 MUSIC 还是 ESPRIT 都可能侵权（等同手段+等同功能+等同效果）
- **查新检索**：必须覆盖所有等同方案，否则漏检（竞品可能用 ESPRIT 而非 MUSIC）
- **语义检索**：等同方案的描述是向量检索的最佳 query——不同结构、同一功能的概念描述让向量模型召回更全

> **操作方法**：对每个必要特征，陆一帆（语义扩展者）须问 3 个问题：
> 1. "这个功能还能用什么不同结构/方法实现？"（列 2-3 种）
> 2. "这些等同方案在专利里各自怎么描述？"（每种给关键词）
> 3. "哪种等同方案最可能被竞品使用？"（标优先检索）

## 三、双路语义扩展（关键词路 + 语义路）

> **关键改进**：每个特征须同时产出两路检索内容——
> - **关键词路**（BM25 精确匹配）：同义词、近义词、英文、分类号
> - **语义路**（向量检索）：自然语言概念描述、功能叙述、问题-效果对

### 关键词路（传统，精确匹配）

| 扩展维度 | 说明 | 示例（以"到达角"为例） |
|---|---|---|
| **同义词** | 不同说法表达同一概念 | 到达角、入射角、angle of arrival、AoA |
| **近义词** | 概念相近但不完全相同 | 到达方向、来波方向、方向角 |
| **上下位** | 更宽/更窄的概念 | 上位：角度测量；下位：MUSIC算法估计AoA |
| **英文对照** | 中文特征对应的英文术语 | angle of arrival, AoA, direction of arrival, DOA |
| **多语种映射** | 日/韩/德文术语（主要专利申请国） | 日：到着角、入射角；韩：도착각；德：Einfalls winkel |
| **IPC/CPC分类号（分层）** | 该特征对应的分类号，标明层级 | 小组：G01S 5/02（精确）；大组：G01S 5/00（宽检）；小类：G01S 5（最宽） |
| **功能表达** | 不用结构词，用功能描述 | 测量信号到达方向、确定信号来源方位 |

### 语义路（向量检索，概念匹配）

> 语义路不是关键词列表，而是**自然语言描述**——让 OpenSearch 的向量模型能"理解"这个特征在说什么。

| 语义维度 | 说明 | 示例（以"AOA角度估计"为例） |
|---|---|---|
| **概念描述** | 一句话说清这个特征在做什么 | "通过测量无线电信号到达天线阵列的角度来估计信号源方向" |
| **问题-效果对** | 这个特征解决什么问题+带来什么效果 | "问题：RSSI信号衰减导致距离估算不准 → 效果：用角度测量替代信号强度，精度提升至厘米级" |
| **类比表达** | 用不同领域的类比描述同一功能 | "类似声源定位中通过麦克风阵列判断声音方向" |
| **否定边界** | 这个特征**不是**什么（帮 OpenSearch 排除噪声） | "不是RSSI信号强度测距，不是飞行时间ToF测距，不是载波相位定位" |

> **为什么语义路重要**：关键词路能找到"说了到达角"的专利；语义路能找到"没说到达角但描述了同样的角度测向原理"的专利。**后者是裸 Claude 检索不到的**——因为裸 Claude 只会想同义词，不会构造概念描述+否定边界。

### 扩展方法

**关键词路**：
1. **同义词**：问"这个特征还能怎么说？"——换主语、换动词、换名词
2. **近义词**：问"有什么概念跟它接近但不一样？"——区分边界
3. **上下位**：问"它的上位概念是什么？它的具体实现有哪些？"——上位扩检索范围，下位缩范围
4. **英文对照**：问"英文专利里这个词怎么写？"——必须查英文，因为大量对比文件是英文
5. **分类号**：问"这个技术领域对应哪个IPC/CPC分类号？"——用分类号锁定技术领域
6. **功能表达**：问"不用具体结构，用功能怎么描述？"——防止对手用不同结构实现同样功能

**语义路**：
1. **概念描述**：问"如果不用任何专业术语，用大白话怎么说这个特征在做什么？"
2. **问题-效果对**：问"这个特征为什么存在？没有它会怎样？有了它怎样？"
3. **类比表达**：问"其他领域有没有做同样事情的不同方法？"
4. **否定边界**：问"容易跟这个特征混淆但实际不同的技术有哪些？"——**这是 sptler 的独家差异化**，裸 Claude 不会主动想"不是什么"，但否定边界能让 OpenSearch 语义检索精准排除噪声。

## 三、检索式构建方法（双路产出）

### 关键词路检索式（给 BM25 用）

#### 1. 核心检索式（先跑这个）
```
(特征A的同义词集合) AND (特征B的同义词集合) AND 技术领域分类号
```

**示例**：
```
(到达角 OR AoA OR angle of arrival OR 方向角) 
AND (蓝牙 OR Bluetooth OR 蓝牙定位) 
AND (IPC=G01S5/02 OR CPC=G01S5/02)
```

#### 2. 扩展检索式（核心无结果时跑）
```
(特征A的上位概念) AND (特征B的功能表达) AND 技术领域
```

**示例**：
```
(角度测量 OR 方向测量) 
AND (短距离无线定位 OR 室内定位) 
AND (IPC=G01S5/00)
```

#### 3. 否定过滤（降噪）
```
核心检索式 NOT (否定边界词)
```

**示例**：
```
(到达角 OR AoA) AND (蓝牙) NOT (RSSI OR 信号强度 OR 载波相位)
```

> **否定过滤是 sptler 的独家**：裸 Claude 不会主动构造 NOT 表达式。但 OpenSearch 的 BM25 支持 NOT，能大幅降低噪声——比如查 AOA 定位时排除所有 RSSI 专利。

### 语义路检索内容（给向量检索用）

> 语义路不是布尔表达式，是**自然语言查询文本**——直接作为向量检索的 query。

#### 1. 语义核心查询
```
{概念描述} + {问题-效果对}
```

**示例**：
```
通过测量无线电信号到达天线阵列的角度来估计信号源方向的方法，
用于解决RSSI信号衰减导致距离估算不准的问题，实现厘米级定位精度。
```

#### 2. 语义扩展查询（用类比拓宽召回）
```
{类比表达} 在专利中的应用
```

**示例**：
```
类似声源定位中通过麦克风阵列判断声音方向的方法，应用于无线通信领域的信号方向估计
```

#### 3. 语义否定边界（帮向量模型排除相似但不同的专利）
```
检索"{概念描述}"但排除"{否定边界描述}"
```

**示例**：
```
检索"无线电信号角度测向定位"但排除"基于信号强度RSSI的测距方法"和"基于飞行时间ToF的测距方法"
```

> **语义路+关键词路双跑**：把关键词检索式的命中 + 语义查询的命中合并去重——这是比单一检索路多出来的召回。**sptler 产出双路内容，下游 OpenSearch 双路跑，合并结果——这是 sptler 比裸 Claude 强的核心：裸 Claude 只产关键词，不产语义查询文本。**

### 4. 场景化检索策略（按场景调精度/召回）

不同场景的检索目标不同，sptler 须按场景调整检索策略：

| 场景 | 检索目标 | 精度/召回 | 策略 |
|---|---|---|---|
| **查新检索** | 找到所有可能破坏新颖性的对比文件 | **高召回**（宁可多查不可漏） | 用上位概念+等同方案+多语种全展开，分类号用大组（G01S 5/00），否定边界放宽 |
| **FTO 检索** | 精确判断是否侵权某具体权要 | **高精度**（避免误判侵权） | 只检必要特征（A+B），用小组分类号（G01S 5/02），等同特征簇重点展开（等同侵权判定），否定边界严格 |
| **无效检索** | 找到能无效特定权要的对比文件 | **精准打击**（针对具体权要） | 逐权要拆特征，每条权要单独构建检索式，从属权要先查（更容易无效），引文追踪优先 |
| **布局检索** | 了解竞争格局和技术空白 | **高召回+趋势** | 上位概念+多语种+发明人/申请人追踪，按时间分布分析技术路线 |

### 5. 检索迭代计划（不是一次性出全部）

> 专利检索不是一次跑完——是**迭代逼近**的过程。sptler 须产出迭代计划，不是一次性检索式。

**第一轮：窄检（高精度）**
- 只用必要特征 A+B 的同义词 + 小组分类号
- 目标：快速找到高度相关的对比文件
- 命中 > 50 → 停，精度够，不需扩
- 命中 < 5 → 进入第二轮

**第二轮：宽检（高召回）**
- 加上位概念 + 等同技术方案 + 多语种 + 大组分类号
- 加语义路查询文本（向量检索拓宽召回）
- 命中检查：高相关率 > 30% → 够，停止
- 高相关率 < 10% → 进入第三轮

**第三轮：精准追踪**
- 从第一/二轮的高相关结果中，做引文追踪 + 同族追踪 + 发明人追踪
- 目标：找到"藏得深"的对比文件（被引频次高的核心专利）
- 这一步是**sptler 比裸 Claude 强的**：裸 Claude 不会主动规划"先窄后宽再追踪"的迭代策略，一次性出全部检索式。

### 6. 补充检索策略
- **引文追踪**：找到1-2篇高相关对比文件后，追踪其引用/被引用
- **同族追踪**：查对比文件的同族（其他国家的对应申请）
- **发明人追踪**：查该领域核心发明人的其他申请
- **申请人追踪**：查竞争对手的专利组合

### 7. 检索完备性自检
完成检索准备后，邹蕴组织自检：
- [ ] 每个必要特征是否都有**关键词路**检索词覆盖？
- [ ] 每个必要特征是否都有**语义路**概念描述+否定边界？
- [ ] **等同特征簇**是否展开（同功能不同结构的替代方案）？
- [ ] **多实施例**是否覆盖（每种实现方式单独检索）？
- [ ] **依赖关系**是否在检索式组合中体现（独立分别检、依赖组合检）？
- [ ] 中英日韩德是否都检索了？
- [ ] 上位概念是否检索了？
- [ ] 分类号是否**分层**覆盖（小组精检+大组宽检）？
- [ ] 否定边界是否构造了（排除噪声）？
- [ ] 是否有特征被遗漏（只检索了A忘了B）？
- [ ] 语义查询文本是否自然通顺（向量模型能理解）？
- [ ] **场景化策略**是否匹配（查新高召回/FTO高精度/无效精准打击）？
- [ ] **迭代计划**是否产出（窄检→宽检→追踪）？
- [ ] 高相关对比文件是否做了引文追踪？

## 四、角色分工（按能力而非固定圣人）

特征分析框架是方法论，不绑定特定圣人。具体由 route_sages 按议题动态路由的入会圣人承担以下角色：

| 角色 | 能力要求 | 职责 |
|---|---|---|
| **特征拆解者** | 擅长结构化拆解技术方案 | 三要素拆解：技术方案→A/B/C/D特征，标注必要性+类型 |
| **边界确认者** | 擅长权利要求边界精修 | 每个特征的边界范围（宽到什么程度、窄到什么程度） |
| **语义扩展者** | 擅长数据语义/检索/分类 | 每个特征展开同义词/近义词/上下位/英文/分类号，构建检索式 |

> 哪位圣人担任哪个角色，由 route_sages 按议题关键词动态路由决定，不在方法论里写死。模板中的"责任人"字段由实际入会圣人填充。

## 五、与其他场景的关系

- **查新检索**：特征拆解→检索准备→用户提供对比文件→逐特征新颖性比对→创造性判断→结论
- **FTO检索**：特征拆解→用户提供风险专利→逐特征侵权比对（全要件+等同）→规避设计
- **无效宣告**：特征拆解→用户提供目标专利→逐特征找无效理由（新颖性/创造性/公开不充分/不支持）
- **OA答复**：特征拆解→审查员引用对比文件→逐特征找区别技术特征→答复策略

特征分析是所有专利场景的共同基础能力。
