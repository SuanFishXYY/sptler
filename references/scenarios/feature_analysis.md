# 专利特征分析框架

> 本文件是 sptler 在查新/FTO/无效等场景进行技术特征拆解与检索内容准备的专业方法。
> 核心思想：**把技术方案拆成可检索的要素，每个要素展开成可执行的检索词，最终构建可跑的检索式。**
> **下游检索引擎**：OpenSearch（语义检索 + 关键词检索双路）。本框架的产出须同时适配两条路。
> **产出格式**：结构化 Markdown 文档（§七 定义章节结构，下游 skill 按标题解析提取）。

---

## 一、三层交叉拆解（权要 × 子特征 × 技术维度）

> **三层交叉是 sptler 比裸 Claude 最核心的拆解差异化**。裸 Claude 只拆到特征级（A/B/C/D），sptler 拆三层：先按权要、再按子特征、再按技术维度——三维交叉，不漏任何可检索角度。

### 1.1 权要级拆解

按权利要求逐条拆解——每条权要独立分析，因为：
- 查新/无效：每条权要单独判断新颖性/创造性
- FTO：每条权要单独判断侵权
- 检索：不同权要的检索式不同（从属权更窄、独立权更宽）

| 权要 | 类型 | 包含特征 | 检索优先级 |
|---|---|---|---|
| 独立权1 | 产品/方法 | A+B | 最高（必须检索） |
| 从属权1 | 优选实施 | A+B+C | 高（从属更易被无效，查新先查从属） |
| 从属权2 | 替代实施 | A+B+D | 中 |
| 从属权3 | 参数收窄 | A+B+E | 低 |

> **从属权先查原则**（无效场景）：从属权比独立权更容易被无效（多了限定特征=多了被对比的可能）。sptler 在无效场景下优先检索从属权——这是裸 Claude 不会主动想到的。

**多条从属权处理**：真实专利可能有 5-10 条从属权，不必每条都做完整三层拆解。按优先级处理：
1. **全部列出**（权要表里列出所有从属权，标含哪些特征）
2. **深度拆解 top 3**：选最窄（最易被无效/最易侵权）的 3 条从属权做完整三层交叉拆解
3. **浅度覆盖其余**：其余从属权只列特征+同义词，不做子特征/维度/等同展开
4. **标注**：深度拆解的标 `[深]`，浅度覆盖的标 `[浅]`

### 1.2 子特征级拆解 + 参数值级

每个特征再拆一层子特征——因为一个"角度估计"可能包含多个可分别检索的技术点。**子特征再往下拆参数值——因为同一个子特征，不同参数值的检索词完全不同**：

| 特征 | 子特征 | 参数值 | 参数值描述 | 独立可检索 |
|---|---|---|---|---|
| A（角度估计） | A1（信号接收） | — | 多天线接收蓝牙信号 | 是 |
| | A2（相位差） | — | 计算天线间相位差 | 是 |
| | A3（角度求解） | 算法=MUSIC | 子空间分解求空间谱峰值 | 是（参数值级） |
| | | 算法=ESPRIT | 旋转不变直接求解 | 是（参数值级） |
| B（天线阵列） | B1（阵元排布） | 间距=λ/2 | 半波长均匀间距 | 是（参数值级） |
| | | 间距=λ/4 | 四分之一波长间距 | 是（参数值级） |
| | B2（阵列校准） | — | 校准阵列相位一致性 | 是 |

> **参数值级是第四层拆解**：裸 Claude 拆到子特征就停了。但竞品可能用 λ/4 间距（不是 λ/2），或用 Root-MUSIC（不是 MUSIC）——**参数值不同=检索词不同=可能找到不同专利**。sptler 对关键参数值单独扩展检索词。
>
> **哪些子特征需要拆参数值**：标"参数"类型的技术维度子特征必须拆；标"结构/方法"的通常不拆（除非有明确数值限定）。
| | B2（阵列校准） | 校准阵列相位一致性 | 是 |

> **子特征拆解的价值**：竞品可能只用了 A1+A2 但用了不同方法做 A3——如果只检索"A 角度估计"整体，会漏掉"只用了不同 A3 方法"的专利。拆到子特征后，A3 可以单独扩展+检索。

### 1.3 技术维度拆解（按技术领域自适应）

> **不是所有专利都有 4 个维度**。软件专利没有材料维度、化学专利结构维度很弱、商业方法没有传统技术维度。sptler 须先判定技术领域，再选该领域有效的维度组合——不硬套 4 维。

**技术领域判定与维度选择**：

| 技术领域 | 判定信号 | 有效维度 | 无效维度（跳过） |
|---|---|---|---|
| **机械/硬件** | 部件、连接、结构、装配 | 材料 + 结构 + 方法 + 参数 | —（全适用） |
| **电子/通信** | 信号、电路、天线、协议 | 结构 + 方法 + 参数 | 材料（通常不关键） |
| **软件/AI** | 算法、模型、流程、数据 | 方法 + 参数 | 材料 + 结构（通常不适用） |
| **化学/材料** | 配方、组分、化合物、比例 | 材料 + 参数 | 结构（分子结构除外）+ 方法（合成方法保留） |
| **生物/医药** | 序列、基因、蛋白、靶点 | 材料（序列/生物材料）+ 方法（实验流程）+ 参数（浓度/剂量） | 结构（3D 结构除外） |
| **商业方法** | 流程、规则、模型 | 方法（业务流程）+ 参数（量化指标） | 材料 + 结构 |

> **自适应逻辑**：sptler 在拆解前先扫议题关键词判定技术领域 → 选有效维度 → 只对有效维度做拆解和扩展。无效维度标"—"不展开，不浪费检索词。**裸 Claude 不会判断领域，硬给 4 维导致软件专利出现无意义的"材料维度"检索词。**

**示例**：
- 蓝牙 AOA 定位（电子/通信）：有效=结构+方法+参数，跳过材料
- React 组件状态管理（软件）：有效=方法+参数，跳过材料+结构
- 锂电正极材料（化学/材料）：有效=材料+参数+方法（合成），跳过结构
- 基因编辑方法（生物）：有效=材料（序列）+方法（实验流程）+参数（浓度），跳过结构

每个子特征在有效维度上的拆解：

| 子特征 | {有效维度1} | {有效维度2} | {有效维度3} | {无效维度} |
|---|---|---|---|---|
| {子特征名} | {维度内容} | {维度内容} | {维度内容} | — |

### 1.4 三要素（问题-方案-效果）

在三层交叉拆解的基础上，仍然保持问题/方案/效果三要素：

- **技术问题**：现有技术的缺陷是什么？（提取问题关键词）
- **技术方案**：上面的三层交叉拆解（权要→子特征→技术维度）
- **技术效果**：带来什么改进？（提取效果关键词+量化指标）

## 二、等同特征簇（同功能不同结构）

> **裸 Claude 会找同义词（换个说法说同一件事），但不会找等同技术方案（换种结构实现同一功能）。** 这是 sptler 的核心差异化。

对每个**子特征**（不只特征级），问：**"还有什么不同的技术手段能实现同样功能？"**

| 子特征 | 功能 | 等同方案 | 方案关键词(供检索) | 方案语义描述(供向量检索) | 竞品优先 |
|---|---|---|---|---|---|
| C1（角度求解） | 由相位差求到达角 | MUSIC算法 | MUSIC, multiple signal classification, 子空间分解, eigen-decomposition, signal subspace | "对天线阵列接收信号的协方差矩阵做特征分解，分离信号子空间与噪声子空间，通过空间谱搜索找峰值估计到达角" | |
| | | ESPRIT算法 | ESPRIT, rotation invariance, 旋转不变, signal subspace rotation, total least squares ESPRIT | "利用天线阵列的平移不变结构，通过子阵间的旋转不变关系直接计算信号方向，无需谱搜索" | ✅(计算量小) |
| | | 2D-FFT波束扫描 | FFT beamforming, 2D-FFT, 波束扫描, spatial DFT, frequency-domain beamforming | "对天线阵列接收信号做二维傅里叶变换，在空间频域找峰值确定信号方向" | |
| A1（信号接收） | 多天线接收信号 | 均匀线阵 | ULA, uniform linear array, 均匀线阵, linear antenna array, 等间距线阵 | "多个天线沿直线等间距排列，同时接收来自不同方向的信号" | ✅(最简单) |
| | | 均匀圆阵 | UCA, uniform circular array, 均匀圆阵, circular antenna array, 圆形阵列 | "多个天线沿圆周等间距排列，提供360度无模糊测向能力" | |

#### 等同方案变体展开

> **每个等同方案可能还有变体**——MUSIC 有 Root-MUSIC/UCA-MUSIC，ESPRIT 有 TLS-ESPRIT/Unitary-ESPRIT。**只搜"MUSIC"会漏掉只写"Root-MUSIC"的专利**。sptler 须对每个等同方案展开 2-3 个变体。

| 等同方案 | 变体 | 变体关键词 | 变体差异 |
|---|---|---|---|
| MUSIC | Root-MUSIC | root-MUSIC, polynomial rooting, 多项式求根 | 用多项式求根替代谱搜索，计算量更小 |
| | UCA-MUSIC | UCA-MUSIC, 圆阵MUSIC, mode-space | 专用于均匀圆阵的MUSIC变体 |
| | 2D-MUSIC | 2D-MUSIC, 二维MUSIC, elevation+azimuth | 同时估计俯仰角+方位角 |
| ESPRIT | TLS-ESPRIT | TLS-ESPRIT, total least squares ESPRIT | 总最小二乘改进，精度更高 |
| | Unitary-ESPRIT | unitary ESPRIT, 西ESPRIT | 利用实值分解降复杂度 |

> **变体展开是 sptler 独家第四层**：裸 Claude 给等同方案就停了。但 MUSIC 的变体 Root-MUSIC 在专利里可能只写"polynomial rooting"不提"MUSIC"——**不展开变体=漏检**。

> **每个等同方案必须给两路关键词**：
> 1. **方案关键词**（BM25 用）：算法名+英文名+技术原理词——下游 OpenSearch 用这些词精确检索采用该方案的专利
> 2. **方案语义描述**（向量用）：一句话描述该方案的原理——让向量模型理解"这种方案在做什么"，能召回没提到算法名但描述了同样原理的专利

等同方案分类（帮 FTO 判断侵权力度）：
- **已知等同**：文献/标准中已记载的替代方案 → 侵权风险高
- **潜在等同**：理论上可行但未见专利记载 → 侵权风险中
- **规避设计**：故意绕开特征的设计 → 侵权风险低，但可能不侵权

> **操作方法**：对每个子特征，语义扩展者须问 3 个问题：
> 1. "这个功能还能用什么不同结构/方法实现？"（列 2-3 种）
> 2. "这些等同方案在专利里各自怎么描述？"（每种给关键词）
> 3. "哪种等同方案最可能被竞品使用？"（标竞品优先）

## 三、双路语义扩展（关键词路 11 维 + 语义路 4 维）

> 每个子特征须同时产出两路检索内容。裸 Claude 只产关键词路。

### 关键词路（BM25 精确匹配，11 维）

| # | 维度 | 说明 | 示例（以"A3 角度求解"为例） |
|---|---|---|---|
| 1 | 同义词 | 不同说法同一概念 | 到达角、入射角、angle of arrival、AoA |
| 2 | 近义词 | 概念相近但不完全相同 | 到达方向、来波方向、方向角 |
| 3 | 上下位 | 更宽/更窄概念 | 上位：角度测量；下位：MUSIC算法 |
| 4 | 英文对照 | 英文专利术语 | angle of arrival, AoA, DOA |
| 5 | 多语种映射 | 日/韩/德（主要专利申请国） | 日：到着角；韩：도착각；德：Einfalls winkel |
| 6 | IPC/CPC分类号（分层） | 标明层级 | 小组 G01S 5/02；大组 G01S 5/00；小类 G01S 5 |
| 7 | 功能表达 | 不用结构词用功能 | 测量信号到达方向 |
| 8 | **共现词** | 该特征在专利里常跟哪些词一起出现 | "天线阵列"+"相位差"+"子空间" |
| 9 | **标准化术语** | 3GPP/IEEE/Bluetooth SIG 等标准怎么描述 | Bluetooth Core Spec: "Angle of Arrival (AoA) based direction finding" |
| 10 | **竞品惯用语** | 高通/华为/三星等在专利里怎么描述 | 高通："spatial direction estimation"；华为："arrival angle measurement" |
| 11 | **缩写/全称变体** | 所有写法变体 | AoA / A.O.A / angle-of-arrival / 到达角测量法 / 到达角估计 |

> **#8-11 是 sptler 独家**：裸 Claude 不会主动想"这个特征在标准里怎么描述"或"高通在专利里怎么写"。但标准术语和竞品惯用语是**精准命中目标专利的关键**——用高通自己的措辞检索，才能找到高通的专利。
>
> **#8 共现词验证**：共现词是基于领域知识推断的，sptler 须标注 `[推断]` 或 `[已验证]`。验证方法：用户提供 1-2 篇已知相关专利文本，sptler 从中提取高频共现词替换推断词。
>
> **#8 共现词推断依据**：标注推断来源，让下游知道可信度：
> - `[推断·领域常识]`：基于该技术领域的通用知识（如"天线阵列"与"相位差"在无线定位领域高频共现）
> - `[推断·类比领域]`：基于相近领域的类比（如从雷达领域推断"子空间"与"MUSIC"共现）
> - `[推断·标准文档]`：从标准文档上下文推断（如 Bluetooth Core Spec 中 AoA 章节常出现"antenna array"+"phase difference"）
> - `[已验证·用户专利]`：用户提供专利文本后提取的实际共现词（最高可信度）
>
> **#10 竞品惯用语验证**：竞品惯用语是基于领域知识预判的，sptler 须标注 `[预判]` 或 `[已验证]`。验证方法：用户指定竞品名称，sptler 建议在第一轮检索结果中按申请人字段过滤，确认该申请人实际使用的措辞。

### 语义路（向量检索，4 维 + 5 准度优化）

> **语义路准度优化的核心问题**：当前查询在同一粒度、缺领域锚定、按单特征查、用大白话、否定太抽象——导致向量模型跨领域误匹配。以下 5 个优化让语义检索"拆得更准"。

#### 4 维基础（保留）

| 维度 | 说明 | 示例 |
|---|---|---|
| 概念描述 | 说清在做什么 | "通过测量无线电信号到达天线阵列的角度来估计信号源方向" |
| 问题-效果对 | 为什么存在+带来什么效果 | "问题：RSSI衰减致距离不准 → 效果：角度测量替代，精度厘米级" |
| 类比表达 | 其他领域同类方法 | "类似声源定位中麦克风阵列判断声源方向" |
| 否定边界 | 这个特征**不是**什么 | "不是RSSI信号强度测距，不是飞行时间ToF，不是载波相位" |

#### 准度优化 1：多粒度查询（粗/中/细三档）

> 向量模型对不同粒度的查询召回不同——粗粒度找回宽但噪声多，细粒度找回窄但漏检。sptler 须给三档，下游 OpenSearch 先跑粗粒度找候选，再用细粒度精排。

| 粒度 | 查询风格 | 示例（C1 角度求解） | 用途 |
|---|---|---|---|
| **粗粒度** | 领域+功能，不提具体技术 | "蓝牙定位中测量信号方向的方法" | 第一轮高召回，找候选集 |
| **中粒度** | 技术+功能，提到关键手段 | "通过天线阵列接收蓝牙信号并计算相位差来估计信号到达角度" | 第二轮平衡，精排候选 |
| **细粒度** | 全部技术细节+参数+效果 | "蓝牙BLE信号通过均匀线阵接收，天线间距半波长，计算天线间相位差，用MUSIC算法求解到达角，实现厘米级室内定位" | 第三轮高精度，确认核心对比文件 |

> **多粒度是 sptler 独家**：裸 Claude 只给一种粒度的语义查询。但向量检索的精度-召回曲线是粒度相关的——三档粒度让下游分阶段调精度。

#### 准度优化 2：上下文锚定

> 裸 Claude 的语义查询"测量信号到达方向"会匹配到雷达/声学/光学领域的方向测量专利——跨领域误匹配。sptler 须在每条语义查询中加**领域锚定词**。

**锚定规则**：每条语义查询必须包含至少 1 个领域锚定词，限定技术领域。

| 锚定层级 | 锚定词 | 排除的误匹配领域 |
|---|---|---|
| 通信领域 | "蓝牙""BLE""无线""radio" | 排除光学/声学方向测量 |
| 定位领域 | "定位""positioning""测向""location" | 排除纯通信信号处理 |
| 应用场景 | "室内""短距离""厘米级" | 排除远程雷达/卫星定位 |

**示例**：
```
无锚定（误匹配）："测量信号到达方向的方法"
  → 匹配到雷达测向、声源定位、光学测角——噪声大

有锚定（精准）："在蓝牙短距离室内定位场景下，通过天线阵列测量信号到达方向的方法"
  → 锚定到蓝牙定位领域，排除雷达/声学/光学
```

> 语义路所有查询（4 维基础 + 多粒度 + 组合查询）都必须带锚定词。**下游 OpenSearch 拿到的语义查询文本自带领域上下文，不需要额外过滤。**

#### 准度优化 3：跨特征组合语义查询

> 当前按单个子特征各给一条语义查询——但有些专利描述了完整系统，每个组件用了不同术语，分开查找不到。sptler 须给一条**组合语义查询**，描述整个发明的完整技术方案。

**组合语义查询模板**：
```
一种{领域锚定}的{发明类型}，包括：{子特征A1的功能描述}；
{子特征B1的功能描述}；{子特征C1的功能描述}；从而实现{技术效果}。
```

**示例**（蓝牙 AOA 定位完整方案）：
```
一种蓝牙短距离室内定位方法，包括：使用多个天线接收蓝牙信号并计算天线间
信号的相位差；由相位差求解信号的到达角度；通过多个角度测量进行三角定位
输出坐标；从而实现厘米级定位精度，替代传统基于信号强度的测距方法。
```

> **组合查询的价值**：能找到"描述了完整 AOA 定位系统但每个组件用了非标准术语"的专利——比如某专利用"方向角推算"代替"到达角估计"、用"多天线接收"代替"天线阵列"、用"坐标解算"代替"三角定位"。单独查每个子特征找不到（术语不匹配），但组合查询的语义描述能让向量模型理解"这个专利在描述同样的系统"。

**组合查询的粒度**：也分粗/中/细三档（同准度优化 1）。

#### 准度优化 4：专利文本风格匹配

> 专利文本有固定行文模式。如果语义查询也用专利行文风格写，向量模型匹配专利文本的相似度更高——因为 OpenSearch 的向量模型训练语料以专利文本为主，风格一致 = 嵌入空间更近。

**专利行文特征**：
- 开头："一种...的方法/装置/系统"
- 中间："包括以下步骤/包含以下部件"
- 特征："其特征在于..."
- 效果："从而实现...""相比于现有技术..."

**风格匹配示例**：
```
自然语言（当前）："通过天线阵列测量信号到达角度来定位"
专利风格（改进）："一种基于蓝牙信号的定位方法，包括以下步骤：
通过天线阵列接收蓝牙信号；计算天线间信号的相位差；
根据所述相位差确定信号的到达角度；根据所述到达角度确定信号源位置。
本发明通过角度测量替代信号强度测距，实现厘米级定位精度。"
```

> **为什么风格匹配更准**：OpenSearch 向量模型在专利语料上训练，"一种...的方法，包括..."这种句式在嵌入空间里与专利文本距离更近。自然语言查询虽然人类能懂，但在向量空间里可能离专利文本较远。sptler 须给**两种风格**——自然语言版（宽召回）+ 专利风格版（高精度），下游可以选。

#### 准度优化 5：负面语义示例

> 当前否定边界只说"不是 RSSI 测距"——但向量模型对"不是"的理解有限。更好的方式是给**负面示例文本**：告诉向量模型"这种文本不该匹配"。

**负面示例模板**：
```
负面示例1（应排除的技术）：{该技术的完整描述}
负面示例2（应排除的技术）：{该技术的完整描述}
```

**示例**：
```
负面示例1："一种基于接收信号强度指示(RSSI)的蓝牙测距方法，
通过信号衰减模型估算发射器与接收器之间的距离，精度为米级。
该方法不涉及角度测量，仅依赖信号强度。"

负面示例2："一种基于超宽带(UWB)脉冲的室内定位方法，
通过测量脉冲飞行时间(ToF)计算距离，使用三边测量法定位。
该方法使用纳秒级脉冲信号，不涉及天线阵列或角度估计。"
```

> **负面示例比否定词更准**：向量模型能理解"这段文本描述的技术与本发明不同"——比"NOT RSSI"这种关键词级否定更有效。下游 OpenSearch 可以用负面示例做**语义排斥**（与负面示例的相似度高的结果降权或排除）。

#### 灰色地带处理（部分相关技术）

> **不是所有技术都能明确"匹配"或"排除"**。如"AOA+RSSI 混合定位"——既不是纯 AOA（应匹配）也不是纯 RSSI（应排除）。sptler 须识别灰色地带并给处理规则。

**灰色地带识别**：sptler 在构造否定边界时，须问——"有没有技术同时包含了本发明特征和否定特征？"

| 灰色地带 | 本发明特征 | 否定特征 | 灰色技术 | 处理规则 |
|---|---|---|---|---|
| AOA+RSSI 混合 | AOA 角度测量 | RSSI 强度测距 | 用 AOA 为主+RSSI 辅助的混合定位 | **不排除，标[灰色]**——保留结果但降权（×0.7），人工复核 |
| AOA+ToF 融合 | AOA 角度测量 | ToF 飞行时间 | 角度+时间融合定位 | **不排除，标[灰色]**——同上 |
| 多天线+单天线切换 | 多天线阵列 | 单天线 | 天线切换（有时多用有时单用） | **不排除**——可能侵权（用了多天线模式时） |

> **灰色地带是 sptler 独家**：裸 Claude 的否定边界是二元的（要么匹配要么排除）。但真实专利有灰色地带——"AOA+RSSI 混合"不能直接排除（它确实用了 AOA），也不能直接当高相关（它也用了 RSSI）。sptler 标 [灰色] + 降权 + 人工复核，比二元排除更专业。
>
> **灰色地带降权公式**：`灰色结果分 = 原始分 × 0.7`（不排除但降权，排在高相关之后、低相关之前）

**负面示例使用方式**（给下游 OpenSearch）：
```
对每条返回结果，计算：
  正面相似度 = cosine(结果向量, 组合语义查询向量)
  负面相似度 = max(cosine(结果向量, 各负面示例向量))
  调整分 = 正面相似度 - 负面相似度 × 0.5
```
> 正面高+负面低 = 真相关；正面高+负面也高 = 可能是相似但不同的技术，需人工复核。

### 语义路扩展方法（更新）

**4 维基础**：概念描述 + 问题-效果对 + 类比表达 + 否定边界（每维给多变体）

**5 准度优化**：
1. **多粒度**：粗/中/细三档查询，下游分阶段调精度
2. **上下文锚定**：每条查询带领域锚定词（蓝牙/无线/定位/室内），排除跨领域误匹配
3. **跨特征组合**：一条完整方案语义查询，找"描述了系统但每个组件用不同术语"的专利
4. **专利风格匹配**：自然语言版（宽召回）+ 专利行文版（高精度），两种风格
5. **负面语义示例**：给应排除技术的完整描述文本，下游做语义排斥（比 NOT 关键词更准）

1-7 见原有方法（同义/近义/上下位/英文/多语种/分类号/功能），新增：

8. **共现词**：问"这个特征在专利文本里通常跟哪些技术词一起出现？"——从已读对比文件或领域常识提取高频共现词，加到检索式提升 BM25 精确度
9. **标准化术语**：问"3GPP/IEEE/Bluetooth SIG 的标准文档怎么定义这个技术？"——标准术语是国际通用描述，能命中跨国专利
10. **竞品惯用语**：问"高通/华为/三星/英特尔在这个领域的专利里用什么措辞？"——用竞品自己的语言检索，精准命中目标专利
11. **缩写变体**：问"这个术语的所有可能写法？"——AoA / A.O.A / angle-of-arrival / 到达角估计——不同写法在 BM25 里是不同 term，全列才不漏

## 四、检索式构建（双路产出 + 场景化 + 迭代计划）

### 4.1 关键词路检索式

#### 核心检索式（窄检，第一轮）
```
(子特征A1的同义词+共现词) AND (子特征B1的同义词) AND 小组分类号
NOT (否定边界词)
```

**依赖链逐级检索式**（不只检 A+C，中间 B 也要检）：
```
第一级(A1): (A1同义词+共现词) AND 小组分类号 → 找到天线阵列相关专利
第二级(A1+B1): 第一级结果 AND (B1同义词+共现词) → 缩小到有相位差计算的专利
第三级(A1+B1+C1): 第二级结果 AND (C1同义词+等同方案) → 精确到角度求解专利
第四级(A1+B1+C1+D1): 第三级结果 AND (D1同义词) → 找到完整定位链的专利
```
> **依赖链检索是 sptler 独家**：裸 Claude 只给一个组合检索式（A AND C），不按依赖关系逐级缩小。但专利检索实战是**逐级过滤**——先找天线阵列专利，再从中有相位差的，再从中有角度求解的。这样每级结果可控，不会一次性出 1000 条噪声。

#### 扩展检索式（宽检，第二轮）
```
(特征A的上位概念+等同方案全部关键词) AND (特征B的功能表达+竞品惯用语) AND 大组分类号
```

**等同方案逐个检索式**（每个等同方案单独跑，不合并）：
```
方案1(MUSIC): (MUSIC OR multiple signal classification OR 子空间分解) AND (蓝牙 OR Bluetooth) AND 分类号
方案2(ESPRIT): (ESPRIT OR rotation invariance OR 旋转不变) AND (蓝牙 OR Bluetooth) AND 分类号
方案3(FFT): (FFT beamforming OR 2D-FFT OR 波束扫描 OR spatial DFT) AND (蓝牙 OR Bluetooth) AND 分类号
```
> **逐个方案检索是 sptler 独家**：裸 Claude 会把所有等同方案 OR 在一起跑一次。但不同方案的关键词不重叠（MUSIC 专利不会写 ESPRIT），OR 在一起会互相干扰。逐个跑更精准，下游合并去重即可。

#### 全称检索式（第三轮，最宽）
```
(技术问题关键词 OR 技术效果关键词) AND 小类分类号 AND 标准化术语
```

#### 否定过滤检索式（降噪，每轮都加）
```
核心式 NOT (否定边界词1 OR 否定边界词2 OR 否定边界词3)
```
**示例**：
```
(到达角 OR AoA) AND (蓝牙) NOT (RSSI OR 信号强度 OR received signal strength)
NOT (飞行时间 OR ToF OR time of flight)
NOT (超宽带 OR UWB OR ultra-wideband)
NOT (载波相位 OR carrier phase)
```
> **否定过滤是 sptler 独家**：裸 Claude 不会主动构造 NOT 表达式。但 OpenSearch 的 BM25 支持 NOT——查 AOA 定位时排除所有 RSSI/ToF/UWB 专利，能把噪声从 1000 条降到 100 条。

### 4.2 语义路检索内容

#### 语义核心查询
```
{概念描述} + {问题-效果对}
```

#### 语义扩展查询
```
{类比表达} 在专利中的应用
```

#### 语义否定查询
```
检索"{概念描述}"但排除"{否定边界}"
```

### 4.3 双路合并策略

> sptler 产出双路内容，下游 OpenSearch 双路跑，**合并去重+按相关度排序**：
> 1. 关键词路结果（BM25 score）+ 语义路结果（向量相似度）合并
> 2. 去重（同一专利在两路都命中 → 提权）
> 3. 按合并分数排序，取 top-N 返回

### 4.4 特征组合新颖性分析

> **裸 Claude 拆完特征就完了——但发明的新颖性常常不在单个特征，而在"特征怎么组合"。** A/B/C 单独都已知，但 A+B+C 的特定组合可能没人做过。sptler 须分析组合新颖性，不只单个特征是否已知。

对每条权要的必要特征组合，做 3 层分析：

**1. 单特征已知性**（已有）：每个子特征单独是否被现有技术公开
**2. 组合新颖性**（新增）：这些子特征的**特定组合**是否被单篇对比文件同时公开
**3. 组合非显而易见性**（新增）：即使组合没被单篇公开，是否存在**结合启示**（两篇对比文件 + 公知常识能否拼出该组合）

| 组合 | 子特征集 | 单特征已知? | 组合被单篇公开? | 结合启示? | 新颖性判断 |
|---|---|---|---|---|---|
| 独权1完整 | A1+A2+B1+C1+D1 | 全已知[推断] | 未知[待检索] | 未知[待检索] | 待定 |
| 核心子组合 | A1+B1+C1 | 部分已知 | 未知[待检索] | 未知[待检索] | 待定 |
| 最小子组合 | A1+C1 | A1已知, C1已知 | 未知[待检索] | — | 待定 |

**子组合逐级分析**（不只检完整组合，还要检所有子组合）：

| 层级 | 组合 | 检索意义 | 检索式 |
|---|---|---|---|
| 完整组合 | A1+B1+C1+D1 | 找公开完整方案的对比文件 | (A1) AND (B1) AND (C1) AND (D1) |
| 核心子组合 | A1+B1+C1 | 找公开核心三特征的对比文件 | (A1) AND (B1) AND (C1) |
| 最小子组合 | A1+C1 | 找公开最小必要组合的对比文件 | (A1) AND (C1) |
| 两两子组合 | A1+B1 | 找公开"接收+相位差"的对比文件（可能用于组合论证） | (A1) AND (B1) |
| | B1+C1 | 找公开"相位差+角度求解"的对比文件 | (B1) AND (C1) |

> **子组合分析是 sptler 独家**：裸 Claude 只检完整组合（A AND B AND C AND D）。但无效检索的核心是**子组合论证**——"对比文件1公开了A+B，对比文件2公开了C+D，本领域技术人员能组合"——需要分别检索每个子组合。不检子组合=无法做组合论证=无效检索失败。

> **组合检索式**（不只是"逐个特征检"，还要"组合检"）：
> ```
> # 单特征检索（逐个检，找公开单个特征的对比文件）
> (A1同义词) AND 分类号 → 找到公开A1的专利
> (C1同义词) AND 分类号 → 找到公开C1的专利
>
> # 组合检索（同时检，找公开组合的对比文件）
> (A1同义词) AND (B1同义词) AND (C1同义词) AND 分类号 → 找到同时公开A1+B1+C1的专利
> ```
> **组合检索是 sptler 独家**：裸 Claude 只做单特征检索，不做组合检索。但新颖性判断的核心是"组合是否被公开"——单特征都已知不代表组合不新颖。

**组合启示线索**（帮无效检索找组合对比文件）：
- 问题线索：对比文件1解决的技术问题 + 对比文件2解决的技术问题 = 能否拼出本发明的组合？
- 功能线索：对比文件1的某特征 + 对比文件2的某特征 = 功能上能否协同？
- 领域线索：同领域的技术人员是否会自然想到这种组合？

### 4.5 检索字段映射

> **不同子特征在不同字段命中率不同**。sptler 须告诉下游 OpenSearch 每个子特征搜哪个字段最准——而不是一刀切搜全部字段。

| 子特征类型 | 最佳搜索字段 | 原因 | OpenSearch 操作 |
|---|---|---|---|
| 结构特征（部件/连接/位置） | **claims** | 权利要求里结构特征描述最精确 | BM25: `claims:(A1同义词)` |
| 方法特征（步骤/流程/顺序） | **description + embodiments** | 方法步骤在说明书实施例里写最详细 | BM25: `description:(B1同义词) OR embodiments:(B1同义词)` |
| 参数特征（数值/范围/阈值） | **examples + claims** | 参数在实施例和从属权里 | BM25: `examples:(B2同义词) OR claims:(B2同义词)` |
| 效果特征（性能/精度/提升） | **abstract + summary** | 效果在摘要和发明概述里最突出 | BM25: `abstract:(效果词)` |
| 问题特征（缺陷/不足） | **background** | 背景技术里描述现有技术缺陷 | BM25: `background:(问题词)` |
| 竞品惯用语 | **applicant + claims** | 按申请人过滤+搜权利要求 | BM25: `claims:(惯用语) AND applicant:(竞品名)` |

#### 权要类型 × 特征类型 交叉字段映射

> **权要类型影响字段选择**：产品权要的结构特征在 claims 最准，但方法权要的方法特征在 description 更详细。sptler 须按权要类型调整字段优先级。

| 权要类型 | 结构特征最佳字段 | 方法特征最佳字段 | 参数特征最佳字段 | 原因 |
|---|---|---|---|---|
| **产品权要** | claims（最准） | claims+description | claims+examples | 产品权要的结构限定在 claims 写最精确 |
| **方法权要** | description（补充） | description+embodiments（最详） | examples+claims | 方法步骤的实施细节在说明书实施例里最详细 |
| **用途权要** | claims | description | — | 用途限定在 claims 里写 |

> **权要类型×字段是 sptler 独家**：裸 Claude 不分权要类型一刀切搜 claims。但方法权要的方法步骤在 claims 里通常只写概括（"计算相位差"），详细步骤（"通过IQ采样提取相位→FFT变换→峰值检测"）在 description 里。**搜错字段=只找到概括描述的专利，漏掉详细实施的专利。**

> **字段映射是 sptler 独家**：裸 Claude 说"搜 title+abstract+claims"一刀切，但结构特征在 abstract 里可能根本没提——搜错了字段等于漏检。sptler 按子特征类型映射到最佳字段，下游 OpenSearch 精准搜对地方。

**字段映射在迭代计划中的应用**：

| 轮次 | 字段策略 | 理由 |
|---|---|---|
| 第一轮（窄） | claims only | 权利要求最精确，先找高相关 |
| 第二轮（宽） | claims + abstract + description | 扩大到说明书，找"没在权要里写但描述了同样技术"的专利 |
| 第三轮（追踪） | full_text + 语义全字段 | 语义检索不限字段，找"用不同语言描述同样原理"的专利 |

### 4.6 结果去重与合并策略

> **3 轮 × 2 路 = 6 组结果**，下游 OpenSearch 返回后必须去重合并。sptler 须给去重规则，不然 6 组重叠结果下游不知道怎么处理。

**去重层级**（从严到宽）：

| 层级 | 去重依据 | 适用场景 | 操作 |
|---|---|---|---|
| L1 专利号去重 | 同一公开号/申请号 | 基础去重（必做） | 相同专利号 → 保留最高分 |
| L2 同族去重 | 同族专利（DOCDB family） | 查新/布局（同族=同一发明不同国家） | 同族只保留1篇（选最早优先权日） |
| L3 发明人去重 | 同一发明人不同专利 | 布局检索（追踪发明人） | 不去重（同发明人不同专利是不同发明） |

**合并评分规则**：

```
最终分 = (BM25分 × 0.4) + (语义相似度 × 0.4) + (组合特征覆盖分 × 0.2)
```

| 系数 | 说明 |
|---|---|
| BM25 分 × 0.4 | 关键词命中度（同义词/共现词/竞品惯用语命中） |
| 语义相似度 × 0.4 | 向量模型打分（概念描述/问题-效果对匹配） |
| 组合特征覆盖分 × 0.2 | 命中了几个必要子特征的组合（全命中=1.0，命中3/5=0.6） |

**双路命中提权**：同一专利在关键词路和语义路都命中 → 最终分 × 1.2（双路确认=更强信号）

**同族自动展开**：
- 高相关结果（最终分 ≥ 3.5）→ 自动查同族，同族结果标 `[同族]` 追加到结果列表
- 中相关结果（2.0-3.5）→ 不自动展开，标注"建议查同族"
- 低相关结果（< 2.0）→ 不展开

> **去重合并策略是 sptler 独家**：裸 Claude 不会给去重规则——但 6 组结果不去重，下游拿到 600 条里 400 条重复，人工筛选崩溃。sptler 给了 3 层去重 + 合并评分 + 同族展开规则，下游拿到的是去重排序后的 top-N。

## 五、场景化检索策略 + 迭代计划

### 5.1 查新检索（高召回——宁可多查不可漏）

**目标**：找到所有可能破坏新颖性/创造性的对比文件

**迭代计划**：
| 轮次 | 检索范围 | OpenSearch 操作 | 返回量 | 终止条件 |
|---|---|---|---|---|
| 第一轮（窄） | 独权必要特征 A+B 同义词 + 小组分类号 | BM25 字段：title+abstract+claims；语义字段：full_text | top 50 | 命中 > 20 且高相关率 > 40% → 停 |
| 第二轮（宽） | +上位概念+等同方案+多语种+大组分类号+竞品惯用语 | BM25 + 语义双跑，合并去重 | top 100 | 高相关率 > 30% → 停 |
| 第三轮（追踪） | 第一/二轮高相关结果的引文+同族+发明人 | 引文追踪（外部查询） | 追加 10-20 | 核心专利被引频次 > 10 → 确认为关键对比文件 |

**否定边界**：放宽（只排除明显不相关的，如查 AOA 定位排除纯 RFID 定位）

### 5.2 FTO 检索（高精度——避免误判侵权）

**目标**：精确判断产品是否落入某具体权要的保护范围

**迭代计划**：
| 轮次 | 检索范围 | OpenSearch 操作 | 返回量 | 终止条件 |
|---|---|---|---|---|
| 第一轮（精准） | 目标权要的必要特征 A+B 精确同义词 + 小组分类号 | BM25 字段：claims（只搜权利要求） | top 30 | 找到包含全部必要特征的专利 → 停 |
| 第二轮（等同） | +等同特征簇全部方案 + 竞品惯用语 | BM25 + 语义双跑，重点搜等同方案描述 | top 50 | 等同方案覆盖完毕 → 停 |
| 第三轮（确认） | 逐条比对命中专利的权利要求 vs 产品特征 | 人工复核（逐特征全要件+等同判定） | 全量 | 每条命中专利完成侵权分析 |

**否定边界**：严格（排除所有非等同的技术方案——如查 AOA 严格排除 RSSI/ToF/UWB）

### 5.3 无效检索（精准打击——针对具体权要）

**目标**：找到能无效特定权要的对比文件

**迭代计划**：
| 轮次 | 检索范围 | OpenSearch 操作 | 返回量 | 终止条件 |
|---|---|---|---|---|
| 第一轮（从属权先） | 从属权（多限定特征）的子特征同义词 + 小组分类号 | BM25 字段：claims+abstract | top 30 | 找到公开全部从属特征的对比文件 → 停 |
| 第二轮（独立权） | 独权的必要特征 A+B + 上位概念 + 大组分类号 | BM25 + 语义双跑 | top 50 | 找到公开全部必要特征的对比文件 → 停 |
| 第三轮（组合） | 用 2-3 篇对比文件组合论证创造性缺乏 | 人工分析（组合对比） | 精选 3-5 | 找到可组合的对比文件组 |

**否定边界**：按权要调整（无效从属权时只排除该从属权独有的特征）

### 5.4 布局检索（高召回+趋势）

**目标**：了解竞争格局、技术空白、技术演进路线

**迭代计划**：
| 轮次 | 检索范围 | OpenSearch 操作 | 返回量 | 终止条件 |
|---|---|---|---|---|
| 第一轮（全景） | 上位概念 + 大类分类号（G01S 全类） | BM25 字段：title+abstract；语义字段：full_text | top 200 | 覆盖主要技术分支 |
| 第二轮（竞品） | 竞争对手名称 + 发明人名称 | 按申请人/发明人字段过滤 | 全量竞品 | 竞品专利组合完整 |
| 第三轮（趋势） | 按时间分布分析第一/二轮结果 | 时间序列分析 | — | 识别技术空白+演进路线 |

## 六、检索覆盖矩阵 + 时间窗口 + 检索日志

> **3 个专业维度**：覆盖矩阵防漏检、时间窗口定范围、检索日志保可复现。裸 Claude 三个都不做。

### 6.1 检索覆盖矩阵

> 36 条自检是平铺列表——看不到"哪个格子还空"。覆盖矩阵把 子特征 × 轮次 × 路径 做成矩阵，一眼看出覆盖盲区。

**矩阵模板**（sptler 须在产出时填写）：

| 子特征 | 第一轮关键词 | 第一轮语义 | 第二轮关键词 | 第二轮语义 | 第三轮追踪 | 覆盖? |
|---|---|---|---|---|---|---|
| A1 | ✅ {检索式编号} | ✅ {粒度} | ✅ | ✅ | ✅ | 全覆盖 |
| A2 | ✅ | ✅ | ✅ | ✅ | — | 全覆盖 |
| B1 | ✅ | ✅ | ✅ | ✅ | ✅ | 全覆盖 |
| C1 | ✅ | ✅ | ✅ | ✅ | ✅ | 全覆盖 |
| D1 | ❌ 漏了 | ❌ | ✅ | ❌ | — | **盲区→补检** |
| E1(从属浅) | ✅ | — | — | — | — | 浅覆盖(OK) |

> **矩阵是 sptler 独家**：裸 Claude 不会画覆盖矩阵——但矩阵是专业检索分析师的标准工具。"哪个子特征在第几轮的第几路没检"一目了然，盲区立刻补检。**漏检的根因是"不知道漏了什么"——矩阵解决了这个问题。**

**覆盖判定规则**：
- 必要子特征（高权重）：关键词+语义两路 × 3 轮 = 6 格，至少 4 格 ✅
- 从属子特征（中权重）：至少 2 格 ✅
- 浅度覆盖子特征（低权重）：至少 1 格 ✅
- 任何必要子特征有 ❌ → **不得产出最终文档，先补检**

### 6.2 时间窗口策略

> 不同场景的检索时间范围不同。sptler 须在迭代计划中指定时间过滤——裸 Claude 不会给时间窗口。

| 场景 | 时间窗口 | 原因 | OpenSearch 操作 |
|---|---|---|---|
| **查新** | 优先权日 **之前** 公开 | 只找优先权日前的现有技术 | `publication_date < {优先权日}` |
| **FTO** | 产品上市日 **之前** 公开 | 只找上市日前已授权/公开的风险专利 | `publication_date < {上市日} AND status = granted` |
| **无效** | 目标专利优先权日 **之前** 公开 | 只找能做 prior art 的对比文件 | `publication_date < {目标专利优先权日}` |
| **布局** | **不限** 或按阶段切分 | 全时间扫描+按技术生命周期分段 | 时间分段：萌芽期/成长期/成熟期 |

> **时间窗口是 sptler 独家**：裸 Claude 不给时间过滤——但不加时间过滤，查新检索会返回优先权日后的专利（无意义），FTO 会返回上市日后的专利（不构成风险）。**时间窗口错了=检索结果全是噪声。**

**时间窗口在迭代计划中的应用**：
- 第一轮（窄检）：加严格时间窗口（如查新只搜优先权日前 10 年）
- 第二轮（宽检）：时间窗口放宽（如查新扩展到优先权日前 20 年）
- 第三轮（追踪）：不限时间（引文追踪可能找到很早的核心专利）

### 6.3 检索日志模板

> 每轮检索该记什么查询/几条结果/几条高相关——**可复现性**。专业检索报告必须含检索日志，裸 Claude 不会主动记录。

**检索日志模板**（sptler 在文档末尾产出，下游 OpenSearch 检索后填回实际数字）：

| 轮次 | 路径 | 查询内容(摘要) | 字段 | 时间窗口 | 预期返回 | 实际返回 | 高相关 | 备注 |
|---|---|---|---|---|---|---|---|---|
| 1 | 关键词 | 核心式(A1+C1同义词+G01S5/02) NOT(RSSI/ToF) | claims | <优先权日 | top50 | {下游填} | {下游填} | {最有效/最无效} |
| 1 | 语义 | 细粒度组合查询(专利行文版) | full_text | <优先权日 | top50 | {下游填} | {下游填} | |

## 七、检索结果质量评估

> sptler 产出后，下游 OpenSearch 返回结果——但结果好不好需要评估标准。sptler 定义评估维度，帮下游 skill 筛选。

### 6.1 相关度评分维度

| 维度 | 权重 | 说明 | 评分标准 |
|---|---|---|---|
| 必要特征覆盖 | 40% | 命中几个必要特征（A+B） | 全命中=5；命中1个=3；未命中=0 |
| 子特征精度 | 20% | 命中的是哪个子特征 | 命中核心子特征（A3）=5；命中辅助（A1）=3 |
| 等同方案匹配 | 20% | 命中的是等同方案还是原方案 | 原方案=5；已知等同=4；潜在等同=3；无等同=0 |
| 技术效果匹配 | 10% | 效果描述是否一致 | 完全一致=5；部分一致=3；不一致=0 |
| 否定边界排除 | 10% | 是否触发否定边界 | 未触发=5；触发=0（直接排除） |

> 合并分数 = Σ(维度分数 × 权重)。总分 ≥ 3.5 = 高相关；2.0-3.5 = 中相关；< 2.0 = 低相关（建议人工复核）。

### 6.2 人工复核要点

高相关结果（≥ 3.5）仍须人工复核：
- [ ] 权利要求全文是否真的包含全部必要特征？
- [ ] 等同方案是否真的等同（手段+功能+效果三重一致）？
- [ ] 优先权日/申请日是否早于目标专利？
- [ ] 是否有同族专利需要一并检查？

### 6.3 迭代终止标准

| 条件 | 动作 |
|---|---|
| 高相关结果 ≥ 5 篇 | 停止迭代，进入人工复核 |
| 三轮跑完仍 < 5 篇高相关 | 停止，标注"检索不充分，建议人工补充" |
| 第二轮后高相关率 < 10% | 可能拆解方向偏差，回到 §一 重新拆解 |

蓝牙BLE信号通过均匀线阵接收，天线间距半波长，计算相位差，用ESPRIT算法求解到达角，实现厘米级室内定位

#### 专利行文版
一种基于蓝牙信号的定位方法，包括以下步骤：通过天线阵列接收蓝牙信号；计算天线间信号的相位差；根据所述相位差确定信号的到达角度；根据所述到达角度确定信号源位置。本发明通过角度测量替代信号强度测距，实现厘米级定位精度。

### 5.4 负面语义示例
