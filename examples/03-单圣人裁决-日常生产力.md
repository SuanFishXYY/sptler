# 示例三：单圣人裁决·日常生产力模式

> 这是 sptler 日常最高频的场景。不是所有问题都需要开议会——一个明确的单领域问题，就该让最相关的 1 位圣人 3 句话给判断。这比裸 Claude 多了"专家人格"，又没有议会仪式的负担。

## 议题

`/sptler 这个权利要求边界怎么收？`

（典型日常问题：单一明确、单领域、无战略信号）

## Phase 1 自动降级

```bash
python scripts/route_sages.py --topic "这个权利要求边界怎么收" --track auto --mode dynamic
```

```
[邹蕴] 路由判定：单圣人裁决（verdict轨：单一明确问题，单圣人裁决），目标 1 人，主持=邹蕴
- [王升] 结构圣｜四核心｜权重 3.5｜关键词命中：边界、权利要求 + 动态加成
```

`--track auto` 识别到：无战略信号、≤1 委员会领域、无中等标记 → 自动降级为 **verdict（单圣人裁决）**，跳过整个议会仪式。

## Verdict 模式执行（3句话，不投票不四律）

召唤王升（带灵魂 + 记忆）：

```bash
python scripts/summon_sage.py --sage 王升 --topic "权利要求边界"
```

若王升有相关记忆则第一句引用；本例王升对"OA流水线"有记忆但对"权利要求边界"无强相关，故作为有经验的圣人发言：

```
[王升] 权利要求边界收口，先把必要技术特征钉死——独立权只留"缺一不可"的A/B/C，
       可扩展的D放到从属权；边界收紧不是加词，是让对手绕不过去。
[王升] 风险点：独权写太宽会被无效，太窄保护范围缩水——用从属权做"退守阵地"。
[邹蕴] 结论已出，记录记忆。议会到此结束。
```

- 不开头脑风暴、不走四律、不投票
- 只更新王升记忆 + 产出权利要求骨架交付物（如适用）

## 交付物（权利要求骨架）

```markdown
# 权利要求骨架 · 权利要求边界收口
> 主笔 王升，精修 卢若雨

## 独立权利要求
1. 一种{技术对象}，其特征在于：
   - {必要特征A}；
   - {必要特征B}；
   - {必要特征C}。

## 从属权利要求（退守阵地）
2. ... {D 进一步限定}
3. ... {替代实施例}

## 边界说明
- 必要技术特征：A、B、C 缺一不可
- 可扩展边界：D 作从属
```

## 四轨制对比：为什么这个问题走 verdict

| 问题 | 自动轨道 | 规模 | 耗时 |
|---|---|---|---|
| "权利要求边界怎么收" | **verdict** | 1人 | 秒级 |
| "OA流水线怎么落地" | fast | 6人 | 1轮 |
| "年度专利战略预算" | formal | 8人 | 完整六阶段 |

sptler 的生产力秘诀就是**按问题重量自动降级**——大多数日常问题落在 verdict，几秒出专家判断；只有真正大的决策才升级到 fast/formal。这样既不会"小问题走大流程"，又保留了重大议题的严谨。

> 议会到此结束。
